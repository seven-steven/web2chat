## 基本信息

- **主要语言:** zh-CN (简体中文)

## 概要

简洁、行动导向的开发者，工作涵盖后端集成、工具开发和部分前端项目。沟通风格以短祈使句为主，决策迅速，独立阅读代码，对工具选型有强烈预设偏好。主要不满触发点：回归缺陷和修复失败。

---

## 维度

### 沟通风格: `terse-direct` (HIGH)

开发者一贯使用简短的中文祈使句，平均不超过 50 词。消息通常是不带铺垫的直接指令（"打 tag"、"翻译成中文"、"修复这个"）。偶尔在提供项目简介或 bug 上下文时会写较长消息，但主导模式在全部 12 个项目中都是 terse-direct。

**Claude 指令:** 匹配简洁风格，保持聚焦和行动导向。不要过度解释。收到任务后立即执行，不复述需求。

**证据:**

- 无铺垫的短指令: "给 git commit 3cc1d91 打上 v1.0 tag" -- 项目: doc-agent
- 最少词汇的片段式祈使: "把所有 skill 都划作 seven-skills 的子 skill" -- 项目: seven-skills
- 简短委托，信任 Claude 自行处理细节: "你自己检查和修正就好" -- 项目: seven-osguider-web

---

### 决策速度: `fast-intuitive` (MEDIUM)

面对选项时快速决策——用"B"加一个微调，或直接委托。没有要求对比表或权衡分析的记录。倾向于迭代调整而非前期深思。偶尔提出针对性的性能问题，但仍然快速做出决定。

**Claude 指令:** 优先给出推荐方案而非列举备选。存在多种方案时，给出推荐 + 简短理由后直接推进，除非开发者重新定向。不要列长对比表，除非明确要求。

**证据:**

- 立即选择并附带简短调整: "B, 同时 changes 改为 baseline" -- 项目: ~/Downloads-test111
- 快速跟进迭代优化: "另外，我希望显式要求 LLM 在 TDD green 之后自动执行 refactor 步骤" -- 项目: ~/Downloads-test111
- 无犹豫的委托: "你自己检查和修正就好" -- 项目: seven-osguider-web

---

### 解释深度: `code-only` (HIGH) — 含覆盖规则

开发者在熟悉领域很少要求解释或讲解。消息聚焦于"做什么"而非"为什么"。提问时精准且有针对性（性能对比、依赖检查），而非概念性问题。直接引用实现细节，表明独立阅读和理解代码。

**覆盖规则（用户明确要求）:** 对于关键设计决策和陌生技术栈，需要概念性讲解——不只是代码。

**Claude 指令:** 常规任务默认直接给代码，跳过概念铺垫。但在做关键架构/设计决策，或涉及开发者不太熟悉的技术栈时，主动提供简洁的概念上下文（这个模式是什么、为什么重要、关键权衡）。保持结构化和精炼——不是教程，但足以建立心智模型。

**证据:**

- 直接委托执行，不要求解释: "你自己检查和修正就好" -- 项目: seven-osguider-web
- Bug 报告 + 直接期望修复: "fix: 文档生成直接失败，具体可参考日志。" -- 项目: doc-agent
- 展示代码理解能力的针对性技术问题: "id desc 和 createTime desc 效果一样，性能是不是更好一点？" -- 项目: 企业后端

### 调试方式: `hypothesis-driven` (MEDIUM)

调试时独立调查并带着具体假设找 Claude。提出假设（"你是不是忘了安装 dayjs 依赖"），修复失败时提供补充数据，提出针对性的诊断问题。也会贴错误日志并隐含期望修复，但表现出对问题领域的认知。

**Claude 指令:** 开发者报告 bug 时，先验证其假设并用证据确认或反驳，再提出修复方案。修复失败后开发者提供补充数据时，优先处理新信息。不要重复解释问题——直接给纠正后的方案。

**证据:**

- 提出具体的根因假设: "你是不是忘了安装 dayjs 依赖" -- 项目: 企业后端
- 修复失败后提供补充数据: "问题没有修复，撤回刚才的提交。补充信息：..." -- 项目: 企业后端
- 基于自身分析的针对性诊断问题: "id desc 和 createTime desc 效果一样，性能是不是更好一点？" -- 项目: 企业后端

### UX 理念: `pragmatic` (LOW)

上下文相关：主要工作是后端集成和工具开发，但在前端项目（osguider-web）中会使用设计工具并提供具体的审美方向。不痴迷设计但也不忽视 UX——务实基线，在前端项目中有 design-conscious 的时刻。

**Claude 指令:** 默认务实的 UX——功能性和整洁，不做花哨设计。开发者给出具体设计方向（风格、审美）时精确执行。

**证据:**

- 带具体审美要求的设计请求: "设计一个高端山地度假村的落地页，风格强调宁静奢华和玻璃拟态效果" -- 项目: ~/Downloads-test111-test
- 使用设计工具 skill 表明设计意识: "使用 /enhance-prompt skill 优化设计 @ux-design-specification.md" -- 项目: seven-osguider-web
- 需要时引用设计文档: "参考 @DESIGN.md 重新生成" -- 项目: seven-osguider-web

### 工具选型理念: `opinionated` (MEDIUM)

对工具有强烈的预设偏好并主动指定。web2chat 项目 CLAUDE.md 包含完整的"不要使用"清单和已选库的明确理由。倾向于构建自定义 skill/plugin 而非使用通用方案。探索新工具时会先研究，但带着具体引用到达。

**Claude 指令:** 尊重已选定的工具偏好。除非有具体技术阻塞，不要建议替代方案。开发者未指定偏好时，推荐最广泛采用的选项并标注为建议。

**证据:**

- 详细的预设技术栈 + 明确排除清单: "WXT 0.20.x, Preact 10.29, @mozilla/readability... 不要使用 Plasmo" -- 项目: seven-web2chat
- 主动指定具体工具并引用参考: "有没有办法把 X 封装成 claude code 的插件" -- 项目: others-exp-land
- 构建自定义 skill 而非使用通用方案: "创建一个 IT 项目研发工时评估的 skill... 先调研网络资料" -- 项目: seven-skills

### 不满触发点: `regression` (MEDIUM)

主要触发点是回归——修复无效或破坏已有功能时立即要求回滚。次要触发点是指令遵循，当 Claude 遗漏步骤时可见（忘装依赖）。多个项目中出现频繁的"用户中断请求"事件，表明开发者在 Claude 偏离方向时倾向于尽早纠偏。

**Claude 指令:** 提交前验证修复有效。改动时明确检查现有功能未受影响。修复失败时立即承认并提出纠正方案，不叠加更多变更。不做超出请求范围的改动——开发者重视有界执行。

**证据:**

- 修复失败后要求回滚: "问题没有修复，撤回刚才的提交" -- 项目: 企业后端
- 指出 Claude 遗漏依赖: "你是不是忘了安装 dayjs 依赖" -- 项目: 企业后端
- 频繁中断表明倾向尽早纠偏: "[Request interrupted by user for tool use]" -- 项目: seven-osguider-web

### 学习风格: `self-directed` (MEDIUM)

自主学习型——独立调查问题，带着精准的针对性问题而非请求通用讲解，引用已经阅读过的外部代码/文档。问题精确（"X 比 Y 快吗？"、"速度差多少？"）而非开放式。

**Claude 指令:** 直接精准回答针对性问题。不要填充开发者未要求的背景上下文。开发者引用外部资源时，针对具体引用展开而非提供通用概述。

**证据:**

- 基于独立分析的针对性性能问题: "id desc 和 createTime desc 效果一样，性能是不是更好一点？" -- 项目: 企业后端
- 具体的事实性问题，非教程: "sata ssd 和 nvme ssd 的速度差多少？" -- 项目: ~/Downloads-test111
- 独立调查后带着具体引用: "有没有办法把 X 封装成 claude code 的插件" -- 项目: others-exp-land
