## Summary

The current Slack adapter already implements two layers of login detection inside `entrypoints/slack.content.ts`: a pathname guard for `/check-login`, `/signin`, and `/workspace-signin`, plus DOM-based login-wall detection via `detectLoginWall()`. [VERIFIED: codebase Read `entrypoints/slack.content.ts`, `shared/adapters/slack-login-detect.ts`] The dispatch pipeline also implements a generic URL remap helper, `isLoggedOutUrlForAdapter()`, which is called from tab-complete, SPA-history, `sendMessage`-failure, and `INPUT_NOT_FOUND` remap branches. [VERIFIED: codebase Read `background/dispatch-pipeline.ts`, `shared/adapters/dispatch-policy.ts`]

The gap is that the generic pipeline helper is currently constrained to `adapter.hostMatches` plus `loggedOutPathPatterns`, and Slack’s registry entry only authorizes and matches `https://app.slack.com/*`. [VERIFIED: codebase Read `shared/adapters/registry.ts`, `shared/adapters/dispatch-policy.ts`, `wxt.config.ts`] If a logged-out Slack navigation lands on a different host such as `slack.com/signin` or `slack.com/check-login`, which Slack’s own sign-in/help surfaces suggest is plausible, the pipeline cannot classify the redirect as `NOT_LOGGED_IN` before injection. [ASSUMED] In that case the service worker proceeds to `chrome.scripting.executeScript()`, and Chrome requires host permission for the page being injected into. [CITED: https://developer.chrome.com/docs/extensions/reference/scripting/] The resulting failure is currently mapped to `INPUT_NOT_FOUND` for permission-related `executeScript` errors, so the user loses the intended login error semantics. [VERIFIED: codebase Read `background/dispatch-pipeline.ts`; CITED: https://developer.chrome.com/docs/extensions/reference/scripting/]

**Primary recommendation:** plan this as a registry/pipeline contract fix plus regression tests; do not touch popup UX, Slack content-script login probes, or message formatting unless a test proves they are part of the failure path. [VERIFIED: codebase Read `entrypoints/popup/App.tsx`, `entrypoints/popup/components/SendForm.tsx`, `entrypoints/slack.content.ts`]

### Pattern 1: Use the pipeline as the first responder for pre-injection redirects

**What:** Classify Slack logged-out redirects before `executeScript()` is attempted. [VERIFIED: codebase Read `background/dispatch-pipeline.ts`]
**When to use:** Whenever the redirected page may be unauthorized for injection or may not host the target editor at all. [CITED: https://developer.chrome.com/docs/extensions/reference/scripting/]
**Example:**
```typescript
// Source pattern: background/dispatch-pipeline.ts + shared/adapters/dispatch-policy.ts
const adapter = findAdapter(record.send_to);
const tab = await chrome.tabs.get(tabId);

if (adapter && tab.url && isLoggedOutUrlForAdapter(adapter, tab.url)) {
  await failDispatch(record, 'NOT_LOGGED_IN', `Redirected to ${tab.url}`, true);
  return;
}
```

## Common Pitfalls

### Pitfall 1: Treating this as a content-script bug

**What goes wrong:** Work starts in `slack.content.ts`, but the failing path never reaches that file. [VERIFIED: codebase Read `background/dispatch-pipeline.ts`, `entrypoints/slack.content.ts`]
**Why it happens:** Slack adapter login probes only run after `executeScript()` and `sendMessage()` succeed. [VERIFIED: codebase Read `background/dispatch-pipeline.ts`, `entrypoints/slack.content.ts`]
**How to avoid:** Start with a pipeline-level failing test that redirects the tab URL away from `app.slack.com` before injection. [VERIFIED: closest reusable pattern in `tests/unit/dispatch/login-detection.spec.ts`]
**Warning signs:** The proposed fix changes Slack selector tests or DOM fixtures but leaves pipeline tests unchanged. [VERIFIED: codebase test layout]

### Pitfall 2: Host-scoped pathname matching silently misses cross-host redirects

**What goes wrong:** `isLoggedOutUrlForAdapter()` returns false because the helper first enforces `isOnAdapterHost()`. [VERIFIED: codebase Read `shared/adapters/dispatch-policy.ts`]
**Why it happens:** Slack registry host scope is only `app.slack.com`, while the logged-out destination may live elsewhere. [VERIFIED: codebase Read `shared/adapters/registry.ts`; ASSUMED]
**How to avoid:** Extend the policy shape deliberately for redirect hosts, or otherwise make cross-host redirect handling explicit in one place. [ASSUMED]
**Warning signs:** A new Slack redirect test still calls `executeScript()` before failing. [VERIFIED: intended regression condition from current code]

### Pitfall 3: Permission-model blind spot

**What goes wrong:** A pipeline URL-based fix is planned without checking whether `tabs.Tab.url` remains visible after a cross-origin redirect under prod permissions. [CITED: https://developer.chrome.com/docs/extensions/activeTab; CITED: https://developer.chrome.com/docs/extensions/reference/tabs]
**Why it happens:** Unit tests run under fake-browser/dev assumptions and cannot prove production permission behavior. [VERIFIED: codebase Read `wxt.config.ts`, `tests/unit/dispatch/login-detection.spec.ts`]
**How to avoid:** Treat this as an explicit planning checkpoint: verify whether current prod permissions can still observe the redirected URL; if not, make a narrow permission-surface decision up front. [ASSUMED]
**Warning signs:** The plan changes redirect matching logic but never mentions permissions or `activeTab`. [CITED: https://developer.chrome.com/docs/extensions/activeTab]

### Pitfall 4: Fixing by widening static permissions without proving necessity

**What goes wrong:** The phase expands `host_permissions` to `https://slack.com/*` by default even if registry/pipeline logic would have been sufficient. [VERIFIED: codebase Read `wxt.config.ts`, `scripts/verify-manifest.ts`]
**Why it happens:** Cross-host redirect bugs tempt implementers to paper over classification gaps with broader permission. [ASSUMED]
**How to avoid:** Make permission widening contingent on a verified inability to read redirect URL state under current permissions. [ASSUMED]
**Warning signs:** The first code diff touches `wxt.config.ts` before a failing pipeline regression test exists. [VERIFIED: current manifest files are unrelated to the bug path]

## Open Questions (RESOLVED)

1. **Does prod-permission runtime still expose the redirected Slack URL to `chrome.tabs.get()` after cross-origin navigation?**
   - Resolution: **No — not reliably under the current production manifest.** Production keeps `permissions = ['activeTab', 'alarms', 'scripting', 'storage', 'webNavigation']` and static `host_permissions` limited to `https://app.slack.com/*`, `https://discord.com/*`, `https://web.telegram.org/*`; it does not ship `tabs`, and it does not ship `https://slack.com/*`. [VERIFIED: codebase Read `wxt.config.ts`, `scripts/verify-manifest.ts`]
   - Supporting evidence: Current Chrome extension docs state sensitive `tabs.Tab.url` access requires `tabs`, matching host permissions, or `activeTab`; and `activeTab` access is revoked on cross-origin navigation. Therefore a redirect from `app.slack.com` to `slack.com` cannot be treated as safely visible under the current production permission surface. [CITED: https://developer.chrome.com/docs/extensions/reference/api/tabs; CITED: https://developer.chrome.com/docs/extensions/develop/concepts/activeTab; CITED: https://developer.chrome.com/docs/extensions/develop/concepts/declare-permissions]
   - Planning consequence: Restoring pre-injection `NOT_LOGGED_IN` for cross-host Slack redirects requires the plan to treat `https://slack.com/*` visibility as a real permission-surface concern rather than an execution-time unknown.

2. **What exact redirect host/path set should Slack declare?**
   - Resolution: The codebase already records the concrete cross-host redirect shape `https://slack.com/check-login` as the critical case. `shared/adapters/slack-login-detect.ts` explicitly states that logged-out Slack users are redirected from `app.slack.com` to `slack.com/check-login`, and `entrypoints/slack.content.ts` still carries same-host login-wall path handling for `/check-login*`, `/signin*`, and `/workspace-signin*`. [VERIFIED: codebase Read `shared/adapters/slack-login-detect.ts`, `entrypoints/slack.content.ts`]
   - Planning consequence: Phase 10.1 should encode a declarative Slack redirect contract that covers at least `https://slack.com/check-login*` plus the existing same-host login-wall paths, and should pair that contract with the smallest manifest verification update needed to observe the cross-host case.
   - Remaining uncertainty: Additional Slack redirect variants may exist, but they are non-blocking for this phase because the known failing cross-host `slack.com/check-login` case is already sufficient to justify the fix boundary and required permission decision.
