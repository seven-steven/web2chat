Output: 一个 2-task TDD 计划，最小触及 shared adapter policy、dispatch pipeline、Slack adapter guardrails 与回归测试；只有在共享 policy 无法单独覆盖 live 路径时，才允许触及 `entrypoints/slack.content.ts` 做同合同对齐。

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| popup → service worker dispatch start | 用户输入的 Slack channel URL 进入 dispatch pipeline，错误语义会被持久化并反馈到 popup。 |
| service worker → browser tab URL state | `chrome.tabs.get(tabId).url` 读取真实导航结果；logged-out redirect 可能落在 `app.slack.com/workspace-signin*` 或 `slack.com/check-login*`。 |
| service worker → content script injection | `chrome.scripting.executeScript()` 之前必须先完成登录态分类，否则权限/注入失败会污染业务错误语义。 |
| shared adapter metadata → dispatch policy | Slack redirect host/path 合同若声明过窄，会再次漏掉 live surface；若声明过宽，会把真实页面访问错误伪装成登录态。 |
| dispatch policy → slack content guard | 若 pipeline 与 Slack content script 对 logged-out 路径理解不一致，live flow 可能在不同阶段产出冲突错误码。 |
| manifest policy → runtime observability | `wxt.config.ts` 与 `scripts/verify-manifest.ts` 必须只保留被真实合同证明需要的 Slack host visibility。 |
