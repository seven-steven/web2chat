## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | inline badge gap、mockup metadata 分隔、icon 与 label 微间距 |
| sm | 8px | chip 内边距、label 与字段值间距、列表紧凑项间距 |
| md | 16px | 正常段落与卡片内边距、按钮水平最小 padding 单位 |
| lg | 24px | section 内内容组间距、双列 mockup 内部间距 |
| xl | 32px | 单个 section 顶部标题与主体内容间距、Hero 文案与 preview 间距 |
| 2xl | 48px | section 垂直 padding 基线、桌面端 Hero 下半区留白 |
| 3xl | 64px | 页面主 section 间距、桌面端首屏上下外边距 |

Exceptions: 44px minimum touch target for CTA buttons, locale toggle, and any icon-only interactive element; 1px border/rule allowed for cards and mockup separators; 2px focus ring uses accent ring token; mobile viewport width under 390px may compress card padding from 24px to 16px only.

**Layout spacing contract:**
- Page shell: `px-6` on mobile, `sm:px-8`, never exceed readable width.
- Hero outer padding: `py-16` mobile, `md:py-16` desktop.
- Non-hero section outer padding: `py-12` mobile, `md:py-16` desktop.
- Section title to section body: 16px.
- Card-to-card gap in compact list sections: 16px.
- Two-mockup desktop pair gap: 24px.
- Footer top border inset aligns with main content width; do not full-bleed the rule unless the whole section is full-bleed.

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 16px | 400 | 1.5 |
| Label | 14px | 400 | 1.4 |
| Heading | 20px | 600 | 1.2 |
| Display | 28px | 600 | 1.15 |

**Typography contract:**
- Use exactly four sizes: 14 / 16 / 20 / 28.
- Use exactly two weights: 400 / 600.
- Hero headline uses Display only once.
- Section headings use Heading only; do not create a larger second display tier.
- Body copy, CTA supporting text, trust copy, limits copy all stay at 16px.
- Labels, badges, mockup metadata, asset status rows, and field names use 14px.
- Payload field values and source/version rows may use mono font, but size must remain 14px or 16px only.

**Per-section type mapping:**
- Hero: 28px display headline + 16px supporting sentence + 14px proof/meta labels.
- Use cases / platforms / flow / CTA section titles: 20px semibold.
- Trust / limits: 20px semibold title, 16px body, 14px risk labels.
- Mockup internals: field label 14px regular, field value 14px regular mono, key emphasis via ink-strong not bold inflation.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | Light `#fafafa`; dark `#1e1e1e` | Page canvas, major background bands, hero backdrop, overall scroll surface |
| Secondary (30%) | Light `#ffffff` / `#f4f4f5`; dark `#262626` / `#2a2a2a` | Cards, mockup panels, supported platform rows, trust/limits surfaces |
| Accent (10%) | Light `#10b981`; dark `#34d399` | Primary CTA, focus ring, step number active fill, send arrow emphasis, selected proof highlight, key source-backed badges |
| Destructive | Light `#dc2626`; dark `#f87171` | Destructive actions only; this page has no destructive interaction |

Accent reserved for: Hero primary CTA, bottom CTA primary button, focus-visible ring, three-step flow connector emphasis on final direction, one “mockup”/“source-backed” badge family, payload send arrow or send target indicator, locale toggle hover underline accent if needed. Do not use accent for every link, every platform badge, whole section backgrounds, or all borders.

**Background alternation contract:**
1. Hero — canvas
2. Use cases — surface-subtle
3. Structured-payload example — canvas
4. Supported platforms — surface-subtle
5. Three-step core flow — canvas
6. Privacy / permissions trust — surface-subtle
7. Known limits — canvas
8. CTA — surface-subtle with optional accent-soft inset panel only around button group

**Semantic status mapping:**
- `--color-warn` / `--color-warn-soft`: Telegram `live UAT pending / known risk`, known-limit chips, caution labels.
- `--color-info`: optional source/version metadata icon accent only; not required.
- `--color-success`: do not use to imply Telegram is fully verified.
- `--color-danger`: forbidden for normal known limits; reserve for true destructive/error semantics only.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | `查看项目源码` |
| Empty state heading | `暂无可展示内容` |
| Empty state body | `当前 section 数据尚未准备好。先补齐 site-content 与对应 i18n 文案，再恢复页面渲染。` |
| Error state | `页面内容加载失败。请检查 locale 文案键、site-content 数据结构和链接配置是否完整。` |
| Destructive confirmation | none — 本 phase 无 destructive action，不设计确认对话框 |

**Narrative contract:**
1. Hero 必须在首屏同时说明三件事：抓取当前网页、保留结构化字段、连同 prompt 投递到 IM / AI Agent 聊天会话。
2. Use cases 只允许 3 个：个人知识沉淀、团队分享、Agent / llm-wiki 工作流。
3. Payload section 必须显式展示 `title` / `url` / `description` / `create_at` / `content` / `prompt` 六类字段。
4. Supported platforms 只展示 OpenClaw / Discord / Slack / Telegram。Telegram 必须带风险标签；Feishu/Lark 不得出现在此 section。
5. Trust section 必须分成 privacy facts 与 permission facts 两组，避免混写成一段大文本。
6. Known limits section 必须说明 Telegram live UAT pending、Feishu/Lark dropped、Phase 11/12 Nyquist partial 为已知风险，不得包装成卖点。
7. CTA section 只给明确去向，不做邮箱订阅、表单收集、waitlist 或 telemetry。

**Hero copy shape:**
- Headline: 一句 value statement，不超过 2 行桌面宽度。
- Supporting sentence: 一句补充，不超过 120 个中文字符或 180 个英文字符。
- Platform chip row: 可出现，但权重低于 headline / CTA / payload preview。

## Section Contract

### 1. Hero
- Width: outer section full-width band; inner content `max-w-4xl` centered.
- Order inside Hero: eyebrow/meta row (optional) → display headline → supporting copy → primary CTA → compact payload preview or popup mockup summary.
- Layout: mobile single column; desktop still single-column overall, but Hero proof area may split into `md:grid-cols-[minmax(0,1.1fr)_minmax(280px,0.9fr)]` only inside the hero content block.
- CTA count: exactly 1 primary CTA in Hero.
- Supporting platform chips: low-contrast inline chips, max 4, no logo wall.
- Do not add navbar, top links, or anchor jumps.

### 2. Use cases
- Width: `max-w-3xl`.
- Layout: 3 compact cards; mobile stacked, desktop 3-column grid if width allows, otherwise 1-column stack with equal card styling.
- Each card contains: title, one-sentence explanation, optional short evidence phrase.
- No decorative illustration required.

### 3. Structured-payload example
- Width: `max-w-3xl`.
- Layout: one primary popup-style mockup panel, not a raw code block.
- Visual contract: field-name column + field-value region; content field may truncate to 5 visible lines with fade or explicit ellipsis.
- Required fields in order: `title`, `url`, `description`, `create_at`, `content`, `prompt`.
- Comparison copy may mention manual copy-paste contrast, but do not create a second competing panel unless space remains generous on desktop.

### 4. Supported platforms
- Width: `max-w-3xl`.
- Layout: compact list or 2x2 grid. Prefer list on mobile, 2-column card grid on `md` and above.
- Required entries: OpenClaw, Discord, Slack, Telegram.
- Telegram row must include warn-toned sublabel: `live UAT pending / known risk` or locale equivalent.
- Feishu/Lark excluded.

### 5. Three-step core flow
- Width: `max-w-3xl`.
- Desktop layout: 3 horizontal steps with arrows/connectors.
- Mobile layout: vertical stack with connectors between steps.
- Step order fixed: ① Capture page → ② Choose target → ③ Send to chat.
- Step marker: circular number badge; final step may receive accent emphasis.
- Copy per step limited to title + one sentence.

### 6. Privacy / permissions trust
- Width: `max-w-3xl`.
- Layout: two grouped cards or stacked lists:
  - Privacy facts
  - Permission facts
- Privacy facts required: user-triggered capture, local storage, direct browser delivery, no remote server, no telemetry, no third-party analytics.
- Permission facts required: production permissions and production host permission model only.
- Dev-only `tabs` and dev-only static `<all_urls>` must not appear as production claims.

### 7. Known limits
- Width: `max-w-3xl`.
- Layout: subdued text list or compact stacked cards; lower visual weight than trust section.
- Required items:
  - Telegram live UAT pending / known risk
  - Feishu/Lark evaluated and dropped from shipped scope
  - Phase 11/12 Nyquist partial as known risk only
- Tone must stay factual, not apologetic, not promotional.

### 8. CTA
- Width: `max-w-3xl`.
- Layout: title + short supporting line + button group.
- Button group: 1 primary (`查看项目源码`) + 1 secondary (README 安装指引).
- Hero CTA and bottom CTA primary button must share identical visual styling.
- Secondary button uses surface styling with border, not accent fill.

## Responsive Behavior

**Breakpoints:** use Tailwind mobile-first defaults; no custom breakpoint system.

- `<640px`: all sections single column; button groups stack vertically if combined width would wrap awkwardly.
- `640px–767px`: keep single-column reading flow; inline chips may wrap to two rows.
- `768px–1023px`: enable 2-column enhancements only for mockup pairings and supported platform grid; flow becomes horizontal if labels still fit within card width.
- `>=1024px`: preserve same narrative order; do not turn page into wide dashboard. Width increase only improves breathing room.

**Non-negotiable responsive rules:**
- Main narrative order never changes across breakpoints.
- No horizontal scrolling.
- Payload mockup and platform cards must remain readable at 320px viewport width.
- CTA buttons may stack on mobile; they must not shrink below 44px height.
- Long URLs in payload example wrap or truncate safely.

## Mockup Labeling Contract

Every proof/mockup module must render a visible status label and a metadata row.

**Visible label:**
- `mockup`

**Metadata row minimum public fields:**
- `source: code-generated`
- `status: marketing demo aligned to current UI contract`
- `version: current repo state` or explicit commit/tag string if executor has it at implementation time

**Hidden from public page but required in implementation comments or data source:**
- owner/update trigger
- capture/creation date

**Mockup coverage:**
- Popup mockup: shows capture result + `send_to` / `prompt` context.
- Target chat mockup: shows delivered message state in chat-like surface.
- Payload example mockup: may reuse popup visual language, but still needs `mockup` label and metadata row.

**Visual style:**
- 1px border with `--color-border-strong`
- card radius `--radius-card`
- surface background only; no gradient-heavy fake screenshot look
- subtle shadow allowed, but keep low-contrast and editorial

## CTA Visual Contract

- Primary button:
  - fill: `--color-accent`
  - hover: `--color-accent-hover`
  - active: `--color-accent-active`
  - text: white in light mode, dark-safe contrast equivalent in dark mode if token mapping changes
  - min height: 44px
  - horizontal padding: 16px minimum, 24px preferred
- Secondary button:
  - background: `--color-surface`
  - border: `--color-border-strong`
  - text: `--color-ink-strong`
  - hover background: `--color-surface-subtle`
- Focus state for both:
  - 2px ring using `--color-accent-ring`
  - not color-only; maintain outline offset or contrast edge
- External-link behavior:
  - visually indicate destination through nearby helper copy if needed, not by cluttering button text
- Link targets:
  - primary → GitHub repository root
  - secondary → README installation section anchor when verified

## Accessibility Acceptance Criteria

Implementation is acceptable only if all of the following are true:

1. Section headings form a valid semantic outline: one `h1`, then section `h2`s.
2. All interactive controls are keyboard reachable in visible order.
3. Focus-visible state is clearly visible on CTA buttons, locale toggle, and any inline links.
4. All decorative mockup shapes are `aria-hidden`; meaningful mockup summaries use visible text, not image-only meaning.
5. Any raster brand/icon image must include correct alt text or empty alt if decorative.
6. Color is not the sole carrier of Telegram risk state; pair warn color with explicit text label.
7. Reduced-motion users do not lose meaning; any motion is optional and guarded by existing token system.
8. Text contrast must remain readable against token surfaces in both light and dark mode.
9. Locale toggle label must be understandable in both locales.
10. Buttons and links preserve minimum 44px target height where expected touch interaction exists.
