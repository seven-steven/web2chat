**Gap 1 fix (blocker) — Two-phase paste injection:**

The root cause: `dispatch-pipeline.ts` injects `discord.content.ts` in `world: 'ISOLATED'`. The `pasteText()` function creates a `DataTransfer` in the ISOLATED V8 context, but Discord's Slate editor in MAIN world reads `clipboardData.getData('text/plain')` and gets empty string (cross-world boundary), producing the "¬" artifact.

Solution: Keep the adapter in ISOLATED world (needed for `chrome.runtime.onMessage`), but delegate the actual paste + Enter to a page-level script injected into MAIN world via inline `<script>` element.

Modify `entrypoints/discord.content.ts`:

1. Remove the `pasteText` function entirely from this file.

2. Add a new function `injectMainWorldPaste(editor: HTMLElement, text: string): Promise<boolean>` that:
   - Creates a `<script>` element with an IIFE that:
     - Listens for a single `message` event on `window` with a specific type (e.g. `{type: 'WEB2CHAT_DISCORD_PASTE', text: string}`)
     - On receiving the message, finds the focused editor element (the ISOLATED script will focus it before posting)
     - Creates a `DataTransfer` in MAIN world context
     - Sets `text/plain` data on it
     - Dispatches `ClipboardEvent('paste', { clipboardData: dt, bubbles: true, cancelable: true })` on `document.activeElement`
     - Dispatches `KeyboardEvent('keydown', { key: 'Enter', bubbles: true, cancelable: true })` on `document.activeElement`
     - Posts back `{type: 'WEB2CHAT_DISCORD_PASTE_DONE'}` to confirm
   - Appends script to `document.documentElement` then removes it (standard MAIN world injection pattern for MV3 extensions)
   - Calls `editor.focus()` (in ISOLATED — this crosses worlds for focus, which works since focus is a browser-level operation)
   - Posts `window.postMessage({type: 'WEB2CHAT_DISCORD_PASTE', text}, '*')`
   - Waits for the `WEB2CHAT_DISCORD_PASTE_DONE` message back (with 3s timeout)
   - Returns true on success, false on timeout

3. In `handleDispatch`, replace the section that calls `pasteText(editor, message)` + Enter keydown dispatch with a single call to `injectMainWorldPaste(editor, message)`. If it returns false, return `{ ok: false, code: 'TIMEOUT', message: 'Paste injection timed out', retriable: true }`.

4. Remove the separate Enter keydown dispatch since it's now handled inside the MAIN world script.

Key constraints:
- The `<script>` element content MUST be a string literal (no external file) since we need it to execute in MAIN world synchronously.
- Use `document.activeElement` in the MAIN script (not a selector) since focus is cross-world.
- The MAIN world script should clean up its message listener after executing once.
- Keep the `chrome.runtime.onMessage` listener in ISOLATED world (it only works there).

**Gap 3 fix (major) — Login detection race condition:**

Root cause: Discord SPA returns HTTP 200 for channel URLs, then client-side JS redirects to `/login`. The existing `onTabComplete` login check runs in `awaiting_complete` state (before the client-side redirect happens). After adapter injection, `chrome.tabs.sendMessage` hangs forever because the content script context gets destroyed by the redirect. The 30s alarm eventually fires as TIMEOUT instead of NOT_LOGGED_IN.

Fix in `background/dispatch-pipeline.ts`, inside `advanceToAdapterInjection`:

1. **Wrap `chrome.tabs.sendMessage` with a 10s Promise.race timeout:**

Replace the current try/catch around `chrome.tabs.sendMessage(tabId, ...)` with:

```typescript
const ADAPTER_RESPONSE_TIMEOUT_MS = 10_000;

let response: { ok: boolean; code?: string; message?: string; retriable?: boolean };
try {
  response = await Promise.race([
    chrome.tabs.sendMessage(tabId, {
      type: 'ADAPTER_DISPATCH',
      payload: { ... },
    }),
    new Promise<never>((_, reject) =>
      setTimeout(() => reject(new Error('ADAPTER_RESPONSE_TIMEOUT')), ADAPTER_RESPONSE_TIMEOUT_MS),
    ),
  ]);
} catch (err) {
  const msg = err instanceof Error ? err.message : String(err);

  // On timeout or sendMessage failure, re-check the tab URL for login redirect
  if (msg === 'ADAPTER_RESPONSE_TIMEOUT' || /Receiving end does not exist|Could not establish connection/i.test(msg)) {
    let actualUrl: string | undefined;
    try {
      const tab = await chrome.tabs.get(tabId);
      actualUrl = tab.url;
    } catch {
      // Tab closed
    }
    if (actualUrl) {
      const adapter = findAdapter(updated.send_to);
      // If URL is now /login or otherwise doesn't match the adapter
      if (adapter && !adapter.match(actualUrl)) {
        await failDispatch(updated, 'NOT_LOGGED_IN', `Redirected to ${actualUrl}`, true);
        return;
      }
    }
  }

  await failDispatch(updated, 'EXECUTE_SCRIPT_FAILED', msg, true);
  return;
}
```

2. **Export the timeout constant** for testability: `export const ADAPTER_RESPONSE_TIMEOUT_MS = 10_000;`

Key constraints:
- Keep the existing error handling for non-timeout sendMessage failures (permission errors etc) — only add the login re-check when the error matches timeout or connection-destroyed patterns.
- The `findAdapter` import is already available at module scope.
- Do NOT change the `world: 'ISOLATED'` line — that stays as-is since the adapter content script needs ISOLATED world for `chrome.runtime.onMessage`.
- The 30s alarm timeout remains as a final safety net in case this 10s check also fails.
