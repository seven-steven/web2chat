# Phase 6 — UI Design Contract

> Visual and interaction contract for Phase 6 (i18n 加固 + 打磨). Generated by gsd-ui-researcher.
> Pre-populated from 06-CONTEXT.md (D-73..D-82), Phase 3 03-UI-SPEC.md (established visual tokens),
> and existing code (`entrypoints/options/App.tsx`, `entrypoints/options/components/`).

Phase 6 introduces **one new UI surface**: the Language selector section on the Options page, replacing
the `ReservedSection` placeholder. All other Phase 6 work is infrastructure (signal-based `t()` rewrite,
i18n coverage script, ESLint hardcoded-string rule) with zero visual changes to existing components.

The existing popup and options page visual language is fully established by Phase 2-5. Phase 6 does not
modify any existing component styling, spacing, or color usage -- it only swaps a grayed-out placeholder
for an active section card with identical visual treatment to existing sections (ResetSection,
GrantedOriginsSection).

## Component Inventory (Phase 6 New)

### LanguageSection

Replaces `ReservedSection` in `entrypoints/options/App.tsx`. Positioned as the **first section card**
below the page heading `<h1>`, before ResetSection and GrantedOriginsSection (D-81: "before all
section cards").

**Structure:**
```
<section class="bg-slate-100 dark:bg-slate-800 rounded-lg p-6 flex flex-col gap-4">
  <header class="flex flex-col gap-2">
    <h2>  -- heading
    <p>   -- explainer
  </header>
  <select>  -- native <select>, 3 options
</section>
```

**Visual treatment:** identical to existing section cards (ResetSection, GrantedOriginsSection):
- Background: `bg-slate-100 dark:bg-slate-800`
- Border radius: `rounded-lg`
- Padding: `p-6` (24px)
- Internal gap: `gap-4` (16px) between header block and select
- Header internal gap: `gap-2` (8px) between heading and explainer

**`<select>` styling:**
```
class="w-full bg-white dark:bg-slate-700
       border border-slate-200 dark:border-slate-700
       rounded-md px-4 py-2
       text-sm leading-normal font-normal
       text-slate-900 dark:text-slate-100
       focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-sky-600"
```

Rationale: matches existing input styling vocabulary (Combobox input border + padding + ring). Native
`<select>` is prescribed by D-80 ("v1 only has two locales, native select is simplest").

**Options (3 total, per D-82):**
1. `Auto (follow browser)` / `自动（跟随浏览器）` -- default
2. `English` -- static label, not translated
3. `简体中文` -- static label, not translated

Option labels for "English" and "简体中文" are intentionally NOT translated (they stay in their
native language regardless of current locale, so the user can always identify the language they want).
Only the "Auto" option label is translated.

**Interaction:**
- `onChange` immediately: (1) writes selected locale to `chrome.storage.local`, (2) updates locale
  signal, (3) entire Options page re-renders in new language within the same render cycle
- No "Save" button needed (D-80: "switch is store + instant effect, no save button")
- Selecting "Auto" clears the stored locale override; locale falls back to `navigator.language` (D-82)

**Accessibility:**
- `<label for="locale-select">` using `t('options_language_label')` -- visible heading doubles as label
- `<select id="locale-select">` with `aria-describedby` pointing to the explainer `<p>`
- Focus ring: `focus-visible:ring-2 focus-visible:ring-sky-600` (Phase 3 established pattern)

**Test IDs:**
- Section: `data-testid="options-language-section"`
- Select: `data-testid="options-language-select"`

## Layout Contracts

### Options Page (updated for Phase 6)

Phase 6 modifies the Options page section order by replacing the `ReservedSection` placeholder.

```
<main class="mx-auto max-w-[720px] p-8 flex flex-col gap-4 font-sans">
  <h1>  -- page heading
  <LanguageSection />        -- NEW (Phase 6, replaces ReservedSection)
  <ResetSection />           -- existing (Phase 3)
  <GrantedOriginsSection />  -- existing (Phase 4)
</main>
```

Position rationale (D-81): Language selector goes first because switching language affects the entire
page. Placing it at top ensures the user sees the immediate effect on all subsequent section cards.

### Popup (no changes)

Phase 6 does not modify popup layout. The signal-based `t()` rewrite causes all existing popup
components to re-render with new translations when locale changes, but no structural or visual
changes occur.

## Interaction Contracts

### Locale Switch Flow

1. User opens Options page (`chrome.runtime.openOptionsPage()` or direct URL)
2. LanguageSection renders with current locale selection (from storage or "Auto" default)
3. User changes `<select>` value
4. `onChange` handler:
   a. If "Auto" selected: clear `locale` key from `chrome.storage.local`
   b. If specific locale selected: write locale value to `chrome.storage.local`
   c. Update locale signal (triggers re-render of all `t()` consumers on the page)
5. Entire Options page immediately reflects new language
6. Next popup open also reflects the persisted locale choice

### Signal Reactivity Boundary

- **Popup + Options (Preact):** signal-based `t()` triggers reactive re-render. All `t()` calls in
  JSX automatically track the locale signal dependency.
- **manifest.json:** `__MSG_*__` placeholders remain controlled by `chrome.i18n` (browser locale).
  Phase 6 cannot change this -- the manifest `name` / `description` / `default_title` follow browser
  language, not the user's runtime override. This is a documented limitation (I18N-04 verifies the
  placeholders exist and resolve correctly).
- **Service worker:** has no UI. Locale changes have no effect on SW behavior.
