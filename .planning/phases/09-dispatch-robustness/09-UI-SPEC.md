# Phase 09 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none — use existing Tailwind v4 token CSS and Preact components |
| Preset | not applicable |
| Component library | none; reuse existing local Preact components (`ErrorBanner`, `InProgressView`, `SendForm`, `ConfirmDialog` pattern) |
| Icon library | inline SVG only; no new icon dependency |
| Font | `var(--font-sans)` system UI stack from `entrypoints/_shared-tokens.css`; mono only for dispatchId pills/host snippets |

Source: existing popup token system in `entrypoints/_shared-tokens.css`, primitives in `entrypoints/popup/components/primitives.tsx`, and Phase 9 Research standard stack (Preact + Tailwind v4, no new packages).

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Inline gaps inside warning/error copy, icon-to-text spacing, focus ring offset equivalents |
| sm | 8px | Button vertical padding, compact row gaps, ErrorBanner retry spacing (`mt-2`) |
| md | 16px | Default section gap/padding for in-progress and confirmation bodies |
| lg | 24px | Dialog/card padding (`p-6`) and major popup vertical spacing |
| xl | 32px | Centered empty/in-progress vertical breathing room (`py-8`) |
| 2xl | 48px | Not used by new Phase 9 popup UI |
| 3xl | 64px | Not used by new Phase 9 popup UI |

Exceptions: existing popup width constraints remain `min-w-[360px]`; icon glyphs remain `16px` for dismiss and `24px` for status/spinner; primary/cancel buttons keep `px-4/px-5 py-2` Tailwind spacing to match current UI.

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 14px | regular 400 | 1.5 |
| Label | 11px | semibold 600 | 1.2 |
| Heading | 14px | semibold 600 | 1.375 |
| Display | 16px | semibold 600 | 1.375 |

Typography contract:

- Use only two weights in this phase: `font-normal` (400) and `font-semibold` (600).
- Retry link, warning confirm action, and primary CTA use 14px/600.
- Low-confidence warning body uses 14px/400; optional technical hint uses 12px/400 if needed.
- Field labels remain 11px uppercase with `tracking-[0.06em]`.
- Do not introduce new font families or 500/700 weights.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | light `--color-canvas #fafafa`; dark `--color-canvas #1e1e1e` | Popup page background and dominant chrome |
| Secondary (30%) | light `--color-surface #ffffff` / `--color-surface-subtle #f4f4f5`; dark `--color-surface #262626` / `--color-surface-subtle #2a2a2a` | Dialog cards, input surfaces, dispatchId pill, subtle disabled button surface |
| Accent (10%) | light `--color-accent #10b981`; dark `--color-accent #34d399` | Primary send/confirm buttons, focus rings, spinner, inline accent words only |
| Destructive | light `--color-danger #dc2626`; dark `--color-danger #f87171` | ErrorBanner left border/text, cancel dispatch button, destructive history/origin confirmations only |

Accent reserved for: `dispatch_confirm_label` button, low-confidence confirmation primary button, focus border/ring, `InProgressView` spinner, inline highlighted action words, and hover state for safe text links. Do not use accent for error recovery retry; retry stays danger-colored in ErrorBanner to preserve current visual language.

Additional semantic color: warning uses light `--color-warn #d97706` / dark `--color-warn #fbbf24` and `--color-warn-soft` only for `SELECTOR_LOW_CONFIDENCE` sending-before-confirmation prompt. Warning color is not destructive and must not replace danger for dispatch failures.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | Existing send CTA remains `dispatch_confirm_label`: en `Send`, zh_CN `投递` |
| Empty state heading | No new Phase 9 empty state; existing history empty state remains en `No history yet — type a URL to start.`, zh_CN `暂无历史记录 — 输入 URL 开始使用。` |
| Empty state body | No new Phase 9 empty body; preserve existing combobox empty-state behavior |
| Error state | Error heading/body continue to be selected by `error_code_<CODE>_heading/body`; retry visibility is controlled by `retriable: true`, not by ErrorCode membership |
| Destructive confirmation | No new destructive action. Existing cancel dispatch remains direct button, no modal; existing history/origin destructive confirmations remain unchanged |

Required new/changed i18n keys for Phase 9:

| Key | en copy | zh_CN copy | Usage |
|-----|---------|------------|-------|
| `selector_low_confidence_heading` | `Confirm target message box` | `确认目标输入框` | Warning prompt heading shown before second dispatch |
| `selector_low_confidence_body` | `web2chat found the message box using a less stable selector. Confirm the target page is correct before sending.` | `web2chat 使用稳定性较低的选择器找到了输入框。发送前请确认目标页面正确。` | Warning prompt body; problem + next step |
| `selector_low_confidence_cancel` | `Cancel` | `取消` | Cancels warning prompt and returns to SendForm without dispatching |
| `selector_low_confidence_confirm` | `Send once` | `仅本次投递` | Primary action; starts a new dispatch with one-shot confirmed flag |

Copy rules:

- All user-visible strings must use `t(...)`; `en.yml` and `zh_CN.yml` must keep 100% key coverage.
- Do not render raw `error.message` from SW/adapter in popup UI.
- Retry labels continue to come from `error_code_<CODE>_retry`; if a retriable code has no specific retry label, fallback label is `Retry` / `重试` via a shared i18n key rather than hard-coded JSX.
- Low-confidence copy must say “confirm before sending”; it must not imply that the first dispatch already sent anything.

## Phase 09 Interaction Contract

### Retriable dispatch error

1. `App.tsx` and `SendForm.tsx` must preserve dispatch error shape as `{ code, message, retriable }` from both immediate `dispatch.start` failure and stored `DispatchRecord.error`.
2. `ErrorBanner` shows its retry button only when `retriable === true` and `onRetry` is provided.
3. ErrorCode controls heading/body/retry copy only; ErrorCode must not decide retry visibility.
4. Clicking retry must:
   - dismiss the current visible error immediately;
   - clear the old active dispatch pointer before creating the new one;
   - build a fresh payload from current `send_to`, `prompt`, `title`, `description`, and `content` form values;
   - use a new `dispatchId`;
   - transition to `InProgressView` immediately after the new active dispatch is created.
5. Retry success follows existing behavior: popup closes after `dispatch.start` returns ok.
6. Retry failure returns to `SendForm` with `ErrorBanner` above all form controls.

### Low-confidence selector warning

1. `SELECTOR_LOW_CONFIDENCE` is a warning, not an ErrorCode.
2. First dispatch that hits a tier3 class-fragment selector must not send the message. It returns/stores a warning that the popup can render.
3. Popup renders a sending-before-confirmation prompt using the existing `ConfirmDialog` visual pattern:
   - overlay: `bg-black/40` light, `dark:bg-black/60` dark;
   - card: `bg-[var(--color-surface)]`, `rounded-[var(--radius-card)]`, `p-6`, `gap-4`, `border-[var(--color-border-strong)]`;
   - heading: 16px/600, body: 14px/400;
   - cancel button: neutral border style;
   - confirm button: accent primary style.
4. Prompt accessibility:
   - `role="dialog"`, `aria-modal="true"`, labelled by heading and described by body;
   - Escape cancels;
   - overlay click cancels;
   - Tab focus stays inside dialog;
   - primary confirm button may auto-focus, matching existing `ConfirmDialog` behavior.
5. Confirming warning starts a second dispatch with:
   - a new `dispatchId`;
   - current form values;
   - one-shot `selectorConfirmation: { warning: 'SELECTOR_LOW_CONFIDENCE' }` or equivalent flag.
6. The confirmation flag must not be saved to history, binding, popup draft, or future sessions.
7. Canceling warning returns to `SendForm` without ErrorBanner and without sending.
8. Reopening popup while a warning is pending must show the confirmation prompt if current capture/form state can be reconstructed; otherwise clear the actionable warning and show the normal SendForm.

### In-progress state

- `InProgressView` remains mutually exclusive with `SendForm` and all warning/error forms.
- Retry and low-confidence confirm both enter `InProgressView` using the same visual contract: centered spinner, heading, body, cancel button, dispatchId pill.
- `aria-live="polite"` stays on in-progress state; ErrorBanner remains `role="alert"` + `aria-live="assertive"`.
