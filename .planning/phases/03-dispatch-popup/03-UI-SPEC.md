# Phase 3 — UI Design Contract

> Visual and interaction contract for Phase 3 (投递核心 + Popup UI). Generated by gsd-ui-researcher,
> verified by gsd-ui-checker. Pre-populated from CONTEXT.md (D-23..D-38) and Phase 2 popup baseline
> (`entrypoints/popup/App.tsx`, `entrypoints/popup/style.css`, `locales/en.yml`, `locales/zh_CN.yml`).

This contract covers three surfaces introduced in Phase 3:

1. **Popup (`entrypoints/popup/`)** — Phase 2 capture preview evolves into SendForm + InProgress + Error views.
2. **Options page (`entrypoints/options/`)** — NEW. Independent URL, opened via `chrome.runtime.openOptionsPage()`.
3. **Toolbar action badge** — `chrome.action.setBadgeText` + `setBadgeBackgroundColor`, three lifecycle states.

The popup is a Preact SPA inside a 360×240 fixed viewport (Phase 2 minimum; Phase 3 may grow vertically
when SendForm + dispatch in-progress views render). The options page is a full-tab page; layout is
single column scoped for v1 with reserved insertion points for Phase 4 (granted origins) and Phase 6
(locale switcher).

**No shadcn. No component library.** Tailwind v4 utility-first only (Phase 1 D-10 locked); component
library was rejected in RESEARCH.md §Standard Stack §Alternatives (Preact target + 360×240 popup
budget make shadcn / radix overhead unjustified). Visual contract below specifies tokens manually.

## Design System

| Property | Value |
|----------|-------|
| Tool | none (Phase 1 D-10 locked Tailwind v4 directly; no component library) |
| Preset | not applicable |
| Component library | none — custom Preact components only (`SendForm`, `Combobox`, `PlatformIcon`, `InProgressView`, `ConfirmDialog`) |
| Icon library | inline SVG (matches Phase 2 `EmptyIcon` / `AlertIcon` pattern in App.tsx; see §Iconography) |
| Font | system UI stack (`font-sans` Tailwind v4 default — no webfont; popup bundle stays small) |
| CSS framework | Tailwind v4 (`@import "tailwindcss";` in `popup/style.css` and `options/style.css`) |

**Why no shadcn:**
1. Project uses **Preact**, not React — shadcn primitives (radix-ui) ship React-only and bring 30+ KB to popup
2. Phase 1 D-10 already locked Tailwind v4 utility-first as the single styling path
3. Phase 3 introduces only one bespoke pattern (combobox) reused twice (send_to + prompt) — `~100 lines` per RESEARCH.md A1, well below the threshold where a library pays off
4. shadcn-style preset would force adoption of tokens that conflict with Phase 2's already-shipped slate/sky/red palette in `popup/App.tsx`

**Reuse from Phase 2:** `textareaClass` constant (focus-visible ring + slate borders), `FieldLabel` component,
inline-SVG icon convention, three-segment i18n split for inline accent spans.

## Spacing Scale

Strict subset of the standard 8-point scale {4, 8, 16, 24, 32, 48, 64}. All Phase 3 NEW component
spacing draws from this set only. All values are Tailwind v4 native scale stops (Tailwind's spacing
unit = 0.25rem = 4px).

| Token | Value | Tailwind | Usage |
|-------|-------|----------|-------|
| xs | 4px | `gap-1`, `p-1` | Icon-to-text gap inside combobox option, badge text padding |
| sm | 8px | `gap-2`, `p-2` | Combobox listbox option vertical padding, ConfirmDialog button row gap, popup chrome title-to-gear gap |
| md | 16px | `gap-4`, `p-4` | Default field stack gap in Phase 3 NEW components (SendForm, InProgressView), popup outer padding (matches Phase 2 `p-4`), error banner padding, options page section gap |
| lg | 24px | `gap-6`, `p-6` | Options page section card padding |
| xl | 32px | `gap-8`, `p-8` | Options page top-level page padding (desktop) |
| 2xl | 48px | — | Reserved (not used in Phase 3) |
| 3xl | 64px | — | Reserved (not used in Phase 3) |

**Exceptions (justified non-grid values):**
- Combobox listbox option **min-height 36px** for keyboard hit target — declared as `min-h-9`
  (Tailwind 9 = 36px). 36px is a 4-multiple but not in the 8-point set; justified because option rows
  must clear the WCAG 2.1 minimum target height while staying compact enough to fit 8 history options
  inside a 360×240 popup without forcing scroll.
- Settings gear icon **24×24** clickable area inside popup chrome (`size-6` = 24px) — 24 is in the
  set, listed for completeness.

**Phase 2 inherited spacing (NOT Phase 3 contract — pre-existing legacy):** the Phase 2 capture preview
already ships with `gap-3` (12px) field stacks and `px-3 py-2` textarea inner padding inside the
existing `textareaClass` constant in `popup/App.tsx`. These are Phase 2 surface; Phase 3 must NOT
modify them (per CLAUDE.md "Surgical Changes" — touch only what you must). Phase 3 NEW components
(SendForm wrapper, Combobox, InProgressView, error banner, options page, ConfirmDialog) use the
strict {4, 8, 16, 24, 32, 48, 64} set above.

**Popup vertical density rationale:** Phase 3 NEW field stacks use `gap-4` (16px) for the SendForm
top region (error banner + send_to + prompt + binding hint + Confirm), then the legacy Phase 2
`gap-3` capture preview region renders below as a self-contained scroll area. Two cadences coexist
because they live on opposite sides of a visual boundary (the divider above the capture preview).

## Typography

Three sizes + two weights only. Pre-populated from Phase 2 popup conventions
(`text-xs leading-snug` labels, `text-sm leading-normal` body, `text-base leading-snug font-semibold` headings).

| Role | Size | Weight | Line Height | Tailwind | Used In |
|------|------|--------|-------------|----------|---------|
| Label | 12px | 400 (normal) | 1.375 (snug) | `text-xs leading-snug font-normal` | FieldLabel above textareas + comboboxes; combobox listbox metadata |
| Body | 14px | 400 (normal) | 1.5 (normal) | `text-sm leading-normal font-normal` | Textarea content, combobox option text, error banner body, dialog body |
| Heading | 16px | 600 (semibold) | 1.375 (snug) | `text-base leading-snug font-semibold` | Empty/Error view heading (Phase 2), InProgressView title, options page section heading |
| Mono (URL) | 12px | 400 (normal) | 1.375 (snug) | `text-xs leading-snug font-mono` | Read-only URL output (Phase 2 keeps this; Phase 3 reuses for send_to history option URL display) |

**Two weights only:** `font-normal` (400) + `font-semibold` (600). No `font-medium` / `font-bold` anywhere.
This matches Phase 2; checker should reject any `font-medium` / `font-bold` introduced in Phase 3 components.

**No display size.** Popup is 360×240; there is no surface large enough to justify a 4th size.

## Visual Hierarchy

**Primary visual anchor (SendForm screen):** the **Send button at bottom-right** — the only filled
sky-600 button on screen, `font-semibold`, right-aligned, with the largest contrast ratio against the
white/slate-900 dominant background. Every other interactive element on the SendForm uses muted
neutrals (slate borders, slate-500 labels) or accent colors confined to focus-visible / active states.
This single high-contrast filled button concentrates the user's attention on the terminal action.

**Primary visual anchor (InProgress screen):** the **24×24 sky-600 spinner + "Sending…" heading**,
center-aligned, font-semibold. The destructive red-600 outline Cancel button is the secondary
affordance, not the anchor.

**Primary visual anchor (Options page):** the page heading + the section card containing the destructive
red-600 "Reset all history" button. Reserved (Phase 4 / Phase 6) cards are visually muted (no actions
yet) and do not compete with the active section.

## Copywriting Contract

Pre-populated from REQUIREMENTS.md DSP-01..10 + STG-03 + ROADMAP.md Phase 3 success criteria.
All strings live in `locales/en.yml` + `locales/zh_CN.yml` under five new namespaces (Phase 2 used flat
underscore-separated keys; Phase 3 continues that style: `dispatch_*`, `error_code_*`, `history_*`,
`binding_*`, `platform_icon_*`, `options_*`).

**Inline accent span pattern (Phase 2 PITFALLS §11 mitigation, MUST be reused):** any string that wraps
an icon or user-supplied URL splits into three keys (`_before` / `_icon` / `_after`); JSX composes
`<>{before}<span class="text-sky-600">{icon}</span>{after}</>`. Never embed HTML in YAML.

**Single-word CTA exception (intentional):** `Send`, `Cancel`, and `Retry` appear as single-word verbs
without an explicit object noun. This is an **accepted exception** because screen context makes the
target unambiguous in every case:
- `Send` (Confirm button) sits below the `Send to` and `Prompt` fields and inherits their object —
  the user has just typed the destination URL into a labeled `Send to` field one line above the
  button. The implicit object is the just-composed message to that destination.
- `Cancel` (in-progress view) appears under the heading `Sending…` with the active dispatchId on
  screen. The object is the visible in-flight dispatch.
- `Retry` (error banner) appears inside the error banner immediately after a labeled error heading
  (e.g. `Timed out` → `Retry`). The object is the failed action just described one line above.
The zh_CN translations (`投递` / `取消` / `重试`) are directionally clear in Mandarin without an
object noun. If Phase 6 i18n polish surfaces ambiguity from translator review, these labels can be
expanded to `Send to platform` / `Cancel dispatch` / `Retry send` at that point.

## Component States

### Combobox (`Combobox.tsx` — used twice for send_to and prompt)

ARIA 1.2 editable combobox with list autocomplete (RESEARCH.md Pattern 5). DOM focus stays in `<input>`;
"active" option indicated via `aria-activedescendant` + visual highlight.

| State | Visual treatment |
|-------|------------------|
| Default (input empty, blurred) | Border `slate-200 / slate-700`; placeholder `text-slate-400`; PlatformIcon shows generic gray globe (24×24) |
| Focus (listbox open) | Border `sky-600 / sky-400`; ring `ring-2 ring-sky-600/20`; listbox renders below with shadow `shadow-lg` |
| Hover (option) | Background `bg-slate-100 dark:bg-slate-800` |
| Active descendant (keyboard nav) | Background `bg-sky-50 dark:bg-sky-950`; left border `border-l-2 border-sky-600`; option text stays slate-900 |
| Selected (after Enter) | Listbox closes; input value updated; PlatformIcon updates to platform-specific icon |
| Disabled (no history yet, prompt before send_to typed) | Text `text-slate-300 dark:text-slate-600`; cursor `not-allowed`; listbox does not open |
| Empty list state (focused but no matching history) | Listbox renders single option with copy `No history yet — type a URL to start.` (text-xs slate-400, no icon, no delete button) |

**Listbox option structure:**

```
┌──────────────────────────────────────────────────────┐
│  [icon]  https://discord.com/channels/123/456    [✕] │  ← 36px height, 8px horizontal padding
│  16×16  text-sm slate-900 truncate            16×16  │     icon + text grow, ✕ delete on hover
└──────────────────────────────────────────────────────┘
```

- Icon: 16×16 PlatformIcon (slate-500 generic if unmatched)
- Text: `text-sm leading-snug font-normal`, truncates with ellipsis at right, `title={fullValue}` for hover-tooltip
- Delete button (✕): 16×16 inline SVG, `text-slate-400 hover:text-red-600`, `aria-label="Remove from history"`,
  visible only on row hover or when option has keyboard focus
- Option vertical padding: `py-2` (8px); horizontal padding: `px-2` (8px); icon-to-text gap: `gap-2` (8px)

**Keyboard contract:**
- `↑` / `↓`: cycle activeIdx through filtered options (-1 → 0 → 1 → … → length−1 → -1)
- `Enter`: select activeOption if activeIdx ≥ 0 (fill input + close listbox); else do nothing
- `Esc`: close listbox, keep current input value (do NOT revert to last selected)
- `Tab`: close listbox, native focus cycle (do not preventDefault)
- `Backspace` on empty input with selected delete-button focused: delete that history entry

**ARIA attributes (verbatim):**
- input: `role="combobox"`, `aria-expanded={listOpen}`, `aria-controls="<id>-listbox"`,
  `aria-autocomplete="list"`, `aria-activedescendant={activeIdx >= 0 ? "<id>-listbox-opt-" + activeIdx : undefined}`
- listbox: `role="listbox"`, `aria-label={fieldLabel}`
- option: `role="option"`, `aria-selected={isActiveDescendant}`, `id="<id>-listbox-opt-{i}"`

## Implementation Notes for Planner / Executor

1. **Reuse Phase 2 primitives:** `FieldLabel`, `textareaClass` constant, inline-SVG icon convention,
   three-segment i18n split. Do NOT re-create — import from existing `popup/App.tsx` (planner should
   extract into `popup/components/primitives.tsx` during Wave 3 to make them options-page-shareable).

2. **Preact attribute naming:** use `for={id}` (not `htmlFor`), `class={...}` (not `className`),
   `stroke-width="2"` (not `strokeWidth={2}`). Phase 2 already locked this; checker should reject any
   React-style camelCase on JSX in Phase 3 components.

3. **No `font-medium` / `font-bold`:** Two weights total (`font-normal` + `font-semibold`). Reject any
   third weight introduced in Phase 3 components.

4. **Spacing scale enforcement:** Phase 3 NEW components use only {4, 8, 16, 24, 32, 48, 64} +
   the documented 36px exception (combobox option `min-h-9`). No `gap-3` / `p-3` / `px-3 py-2` /
   `gap-5` / `p-5` in any new file. Phase 2 inherited code keeps its `gap-3` and `px-3 py-2` (in
   `textareaClass`); do not touch them.

5. **Badge color values are hardcoded hex** in `chrome.action.setBadgeBackgroundColor()` calls (the API
   accepts hex strings, not Tailwind class names). Keep them in a single `BADGE_COLORS` constant in
   `background/dispatch-pipeline.ts`:

   ```ts
   const BADGE_COLORS = {
     loading: '#94a3b8', // slate-400
     ok: '#22c55e',      // green-500
     err: '#ef4444',     // red-500
   } as const;
   ```

6. **Popup mount logic order** (replaces Phase 2's single capture.run dispatch):

   1. Read `chrome.storage.session.get('dispatch:active')` → if record exists with state ∉ {done, error, cancelled},
      render `InProgressView` (do NOT trigger capture.run)
   2. Else, run Phase 2's `capture.run` RPC AND read `popupDraft` storage.local item in parallel
   3. Read `last_dispatch_error` from storage.local (D-34 err badge persistence) → if set, render error banner
      above SendForm; popup mount also clears `chrome.action.setBadgeText({text:''})` per D-34
   4. Render SendForm + capture preview

7. **i18n key namespace conventions:**
   - `dispatch_*` — all dispatch-flow copy (in-progress, confirm button, cancel)
   - `error_code_*` — five new ErrorCodes; subkeys `_heading` / `_body` / `_retry` (where applicable)
   - `history_*` — combobox listbox copy (empty state, View all, delete tooltip)
   - `binding_*` — soft-overwrite link (three-segment: `_before` / `_icon` / `_after`)
   - `platform_icon_*` — platform tooltip alt text (`_mock`, `_openclaw`, `_discord`, `_unsupported`)
   - `options_*` — options page copy (heading, reset section, confirmation dialog, toast)
   - `popup_chrome_*` — popup title + settings gear tooltip

8. **Conditional rendering:** SendForm and InProgressView are mutually exclusive. Error banner is always
   rendered conditionally above SendForm; never above InProgressView.

9. **Empty / Error / Loading views from Phase 2:** kept as-is for the capture flow. Phase 3 adds
   InProgressView + Error banner only; does NOT modify the existing 4 capture states.
