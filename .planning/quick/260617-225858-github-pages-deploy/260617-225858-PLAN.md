# Quick Task 260617-225858: 添加 GitHub Actions 配置，把 webpage 发布到 GitHub Pages

## 目标

把 `apps/marketing`（web2chat 宣传页，pnpm workspace 子包）通过 GitHub Actions 自动构建并发布到 GitHub Pages，部署形态为**项目子路径** `https://seven-steven.github.io/web2chat/`，**仅在 push 到 main 时触发**。

## 已验证的关键事实

- webpage 位于 `apps/marketing/`，Vite 构建，产物在 `apps/marketing/dist`。`index.html` 用绝对根路径引用 `/favicon.svg`、`/src/main.tsx`。
- 仓库 `git@github.com:seven-steven/web2chat.git`，默认分支 `main`。GitHub Pages 项目站点 URL 必为 `https://seven-steven.github.io/web2chat/`（子路径 `/web2chat/`）。
- 本地实测：`vite build --base=/web2chat/` 后产物内 favicon 变 `/web2chat/favicon.svg`、script/css 变 `/web2chat/assets/...`，且 `dist/favicon.svg` 仍在 dist 根 → base 注入正确，资源路径与子路径自洽。
- 现有 `ci.yml` 已有完整 pnpm 设置链路（`actions/checkout@v6` + `pnpm/action-setup@v6` + `setup-node@v6` + `frozen-lockfile`）+ `site:build`/`site:verify` gate。
- **约束（已确认零冲突）**：`apps/marketing/scripts/verify-build.mjs:140` 硬性正则校验 `index.html` 里 favicon 必须是 `/favicon.svg`（绝对根路径），且 `dist/favicon.svg` 须存在于根。该 gate 仅在普通 `site:build`（base `/`）后由 CI `ci.yml` 与本地跑——本方案部署 job 用环境变量注入 `/web2chat/` base，**不修改 verify-build.mjs**，故 CI 主验证链路完全不受影响。

## 设计决策

### D-1: base path 经环境变量注入，不硬编码

`apps/marketing/vite.config.ts` 改为：

```ts
base: process.env.VITE_BASE ?? '/',
```

- 默认（本地 dev、CI `site:build`/`site:verify`、`site:dev`/`site:preview`）：base `/`，行为与现状完全一致。
- 部署 job：`VITE_BASE=/web2chat/ pnpm site:build`，产物路径带上子路径前缀。

**为什么不改 verify-build.mjs**：主验证链路（`pnpm site:build && pnpm site:verify`）校验的是 base `/` 产物，favicon 正则匹配 `/favicon\.svg` 不变。部署产物用 `/web2chat/` base 是独立路径，不经过 `site:verify`，零冲突、零回归风险。

### D-2: 新建独立 deploy workflow，不污染 ci.yml

新建 `.github/workflows/deploy-pages.yml`，与现有 `ci.yml`（验证）/`release.yml`（发布 zip）职责分离。Deploy workflow 仅在 push 到 main 时运行。

### D-3: 用官方 actions，双 job（build → deploy）

遵循 GitHub 官方 Pages 部署标准：
- `build` job：checkout + pnpm install + `VITE_BASE=/web2chat/ pnpm site:build` + `actions/upload-pages-artifact@v3`（path: `apps/marketing/dist`）。
- `deploy` job：`needs: build`，`environment: github-pages`，`permissions: { pages: write, id-token: write }`，`actions/deploy-pages@v4`。

> 注：upload-pages-artifact 当前最新稳定主版本 v3 / v1 均见于社区，executor 落地时按最新可用 major pin（优先 `@v3`，若不可用退 `@v1`），并在 SUMMARY 标注实际采用的版本。

## 任务清单

### Task 1: vite.config.ts 支持 base 环境变量

- **files:** `apps/marketing/vite.config.ts`
- **action:** 在 `defineConfig({...})` 顶层加 `base: process.env.VITE_BASE ?? '/'`（放在 `plugins` 之前或之后均可，符合现有字段顺序）。
- **verify:**
  - `cd apps/marketing && pnpm exec vite build` → 产物 favicon 仍为 `/favicon.svg`（默认 base 生效）。
  - `VITE_BASE=/web2chat/ pnpm exec vite build` → 产物 favicon 为 `/web2chat/favicon.svg`。
- **done:** 两种 base 均按预期产出，默认行为不变。

### Task 2: 新建 deploy workflow

- **files:** `.github/workflows/deploy-pages.yml`
- **action:** 创建 workflow：
  - `on: push: branches: [main]`，`on: workflow_dispatch:`（保留手动重发布能力，零成本）。
  - 并设 `permissions: { contents: read }`（job 级再细化）。
  - `build` job：`runs-on: ubuntu-latest`，`timeout-minutes: 10`，steps:
    1. `actions/checkout@v6`
    2. `pnpm/action-setup@v6`
    3. `actions/setup-node@v6`（node 20，`cache: pnpm`）
    4. `pnpm install --frozen-lockfile`
    5. `VITE_BASE=/web2chat/ pnpm site:build`（env 注入 base）
    6. `actions/configure-pages@v5`
    7. `actions/upload-pages-artifact@v3` with `path: apps/marketing/dist`
  - `deploy` job：`needs: build`，`runs-on: ubuntu-latest`，`environment: { name: github-pages, url: ${{ steps.deployment.outputs.page_url }} }`，`permissions: { pages: write, id-token: write }`，单 step `actions/deploy-pages@v4`（id: deployment）。
- **verify:** YAML 语法校验（`pnpm exec yaml` load 或 python yaml）；本地 dry-run 构建路径已实测。
- **done:** workflow 文件语法合法，引用的 action 版本均为官方维护版本，结构与官方 Pages 部署文档一致。

### Task 3: 本地回归 + 文档

- **files:** 无（验证为主），可选在 `apps/marketing/` README 或顶层 README 补一句部署说明。
- **action:**
  - 跑 `pnpm typecheck`、`pnpm lint`、`pnpm site:build && pnpm site:verify` 确认主链路无回归（vite.config 改动只增一个环境变量读取，不影响类型/lint）。
  - 确认 `pnpm-workspace.yaml`、`package.json` 无需改（`site:build` 已存在）。
- **verify:** 全部 gate 通过。
- **done:** 工作区改动 = 1 个 vite.config 行 + 1 个新 workflow 文件；CI 主验证链路绿。

## 风险与边界

- **需用户在 GitHub 仓库 Settings → Pages → Source 设为 "GitHub Actions"**（首次部署前提）。部署 job 失败时报错会提示环境未启用——这是预期，属仓库侧一次性配置，不在代码范围内。SUMMARY 会显式提示。
- upload-pages-artifact version pin：执行时按实际可用 major 落地并标注。
- 不触碰扩展主链路代码、不改 host_permissions、不改 manifest。
