## v1 Requirements

Requirements for v1.2. Each maps to roadmap phases.

### Positioning & Messaging

- [x] **MSG-01**: Visitor can understand from the hero section that web2chat sends the current web page's structured information plus a saved prompt to an IM or AI Agent chat session. — ✓ **Validated in Phase 15** (hero one-line value + structured-payload preview, content truth test CLM-HERO-01 from Phase 13)
- [x] **MSG-02**: Visitor can identify the primary use cases: personal knowledge capture, team sharing, and Agent / llm-wiki workflows. — ✓ **Validated in Phase 15** (use cases section, content truth test CLM-USE-01 from Phase 13)
- [x] **MSG-03**: Visitor can understand why web2chat is different from manual copy-paste through a concise structured-payload example. — ✓ **Validated in Phase 15** (payload example section with locked field order, content truth test CLM-PAYLOAD-01 from Phase 13)

### Product Proof

- [x] **PROOF-01**: Visitor can see the currently shipped platform set as OpenClaw, Discord, Slack, and Telegram. — ✓ **Validated in Phase 15** (platform section whitelist of 4 shipped platforms + verify-claims rule enforcing `shipped-platforms.json`)
- [x] **PROOF-02**: Visitor can follow the core three-step flow: capture page, choose target, send to chat. — ✓ **Validated in Phase 15** (Stepper `<ol>` with compile-time 3-step tuple + TargetMockup)
- [x] **PROOF-03**: Visitor can see credible product evidence through screenshots, diagrams, or UI mockups that are labeled with their source/version status. — ✓ **Validated in Phase 15 + Phase 16** (PopupMockup / TargetMockup with visible `mockup` label + source/status/version AssetLabel; verify-claims rule enforces all 4 proof.* keys + `proof.label==='mockup'`)

### Trust & Constraints

- [x] **TRUST-01**: Visitor can understand the privacy model: user-triggered capture, local-first storage, direct browser delivery to the chosen chat page, no telemetry, and no third-party analytics. — ✓ **Validated in Phase 15 + Phase 16** (trust section content truth test CLM-PRIVACY-01; verify-claims forbids cloud sync / user tracking wording)
- [x] **TRUST-02**: Visitor can understand the production permission model without dev-only or misleading permission claims. — ✓ **Validated in Phase 15 + Phase 16** (content truth test CLM-PERM-01; verify-claims cross-checks production `wxt.config.ts` manifest)
- [x] **TRUST-03**: Visitor can distinguish shipped capabilities from known risks and deferred platforms, including Telegram live UAT and Feishu/Lark scope status. — ✓ **Validated in Phase 13 + Phase 16** (content truth CLM-PLATFORM-01/CLM-LIMIT-01/CLM-LIMIT-02; CHANGELOG [v1.2] Known Issues with 3 honest boundary items)

### Acquisition CTA

- [x] **CTA-01**: Visitor can find the primary source repository or project entry point from the page. — ✓ **Validated in Phase 15** (hero + footer CTA wired to REPO_URL, G201 external-link indication)
- [x] **CTA-02**: Visitor can find a clear installation or availability path for Chrome / Chromium users, even if the current state is an install-from-source or release placeholder. — ✓ **Validated in Phase 15** (footer-secondary CTA → README `## 安装` anchor + `verify:readme` guard against anchor drift)

### Content Operations

- [x] **OPS-01**: Maintainer can update the platform list, privacy claims, screenshots, and CTA text from explicit source sections/files without hunting through implementation details. — ✓ **Validated in Phase 13 + Phase 16** (maintenance rules in 13-CONTENT-SOURCES.md; root-level `MAINTENANCE.md` with source-first → artifact-second → page-last chains)
- [x] **OPS-02**: Maintainer can verify that promotional claims match `PROJECT.md`, `PRIVACY.md`, `STORE-LISTING.md`, and production `wxt.config.ts`. — ✓ **Validated in Phase 13 + Phase 16** (verification checklist in 13-CONTENT-SOURCES.md; `verify:claims` cross-source consistency verifier as CI gate)

### Build & Deploy Isolation

- [x] **BUILD-01**: Developer can build the promotional page through a dedicated static-site command without changing the extension build output. — ✓ **Validated in Phase 14 + Phase 16** (`site:build` produces `apps/marketing/dist/`, extension output `.output/chrome-mv3/` unchanged)

- [x] **BUILD-03**: Promotional page code does not import extension runtime modules such as service worker logic, storage repositories, messaging, permissions, or IM adapters. — ✓ **Validated in Phase 14 + Phase 16** (`marketing-isolation.spec.ts` source-scan regression test passes)

## Out of Scope

Explicitly excluded. Documented to prevent scope creep.

| Feature | Reason |
|---------|--------|
| Modifying capture or dispatch pipeline | v1.2 is a promotional page milestone, not extension runtime work |
| Telegram live UAT closeout | Recorded as known risk only; separate verification work if prioritized later |
| Phase 11/12 Nyquist closeout | Recorded as known risk only; separate validation work if prioritized later |
| Feishu/Lark adapter revival | Dropped from shipped scope after v1.1; not part of promotion page delivery |
| Full docs portal / blog / CMS | Larger website scope; defer until the static landing page proves useful |
| Newsletter / lead capture / telemetry | Conflicts with minimal static scope and local-first trust posture |
| Server-side rendering or backend services | Static page is sufficient for v1.2 and avoids operational complexity |
| Marketing claims for unshipped platforms | Would misrepresent shipped scope and create trust risk |
