# Phase 2 — UI Design Contract: 抓取流水线 (Capture Pipeline)

> Visual + interaction contract for the **popup capture surface**. The popup is the entire UI surface for this phase: it auto-fires `capture.run` on mount and renders one of three mutually exclusive states (loading / empty / error / success). Generated by gsd-ui-researcher; verified by gsd-ui-checker.

**Phase 2 popup scope (locked by 02-CONTEXT.md D-13..D-22):**
- popup mount → auto fires `capture.run` RPC (no Capture button, D-15)
- 4 view states: `loading` / `empty` / `error` / `success` (mutually exclusive)
- success state: 5 fields displayed; `title` / `description` / `content` editable (D-21), `url` / `create_at` read-only
- editing values live only in Preact signals — no draft persistence in Phase 2 (D-22)
- all user-visible copy via `t('capture.*')` namespace; en + zh_CN 100% isomorphic (PITFALLS §11 + Phase 1 cross-cutting constraint)
- **out of scope:** send_to / prompt / history / send button / settings (Phase 3+)

## Design System

| Property | Value |
|----------|-------|
| Tool | none (manual Tailwind v4 utilities only) |
| Preset | not applicable |
| Component library | none (Preact 10.29 + native HTML elements only) |
| Icon library | none (inline SVG glyphs in JSX; ≤2 needed for empty + error) |
| Font | system stack via `font-sans` (Tailwind v4 default `ui-sans-serif, system-ui, ...`) |

**Why no shadcn / no Radix:** this is a Preact (not React) Chrome popup with a hard <100KB bundle budget. shadcn primitives are React-only and pull `react-dom` + `@radix-ui/*`. CONTEXT.md locks Preact 10.29 + signals; vendor decision is `opinionated` per developer profile. Native `<textarea>`, `<button>`, `<output>`, `aria-busy`, `aria-live` cover every contract this phase needs.

**Color scheme:** popup respects `prefers-color-scheme` via the `dark:` Tailwind variant + `color-scheme: light dark` already set in `entrypoints/popup/style.css`. Both light and dark are first-class — popup chrome must not assume either.

## Layout & Sizing

| Property | Value | Rationale |
|----------|-------|-----------|
| popup body min-width | **360px** | Chrome popup default range 360–400px; 360 is the sweet spot for content readability without forced scroll on narrow screens. Phase 1 baseline `min-w-[280px]` is **upgraded to `min-w-[360px]`** for Phase 2 to fit the always-on textarea field labels comfortably. |
| popup body max-width | **400px** | Chrome popup soft cap; beyond this the popup looks like a panel. |
| popup body min-height | **240px** | Phase 1 baseline `min-h-[120px]` is **upgraded** so loading skeleton + empty / error states do not cause layout shift to success state. |
| popup body max-height | **600px** | Below typical viewport. Content beyond 600px scrolls inside `<main>` (the `<textarea>` for `content` is the long region). |
| Outer padding | `p-4` (16px) all sides | Single padding token covering states. |
| Vertical rhythm between fields | `gap-3` (12px) | 12px is multiple-of-4; tighter than 16 because labels stack tightly above textareas. |
| Vertical rhythm inside a field (label → textarea) | `gap-1` (4px) | xs spacing. |

Exceptions: none.

## Typography

3 sizes, 2 weights — within the prescriptive 3–4 size / 2 weight cap.

| Role | Size (px) | Weight | Line Height | Tailwind utility | Usage |
|------|-----------|--------|-------------|------------------|-------|
| Body | 14 | 400 (regular) | 1.5 | `text-sm leading-normal font-normal` | textarea content, read-only field values, error / empty body copy |
| Label | 12 | 400 (regular) | 1.4 | `text-xs leading-snug font-normal` | field labels above each textarea, read-only field labels — distinguished from body purely by size + muted color |
| Heading | 16 | 600 (semibold) | 1.3 | `text-base leading-snug font-semibold` | empty state heading, error state heading |

**Font family:** `font-sans` (Tailwind v4 default `ui-sans-serif, system-ui, -apple-system, "Segoe UI", Roboto, "Helvetica Neue", Arial, "Noto Sans", sans-serif, "Apple Color Emoji", "Segoe UI Emoji"`). Stack covers en + zh_CN (Noto Sans + system Chinese fallback) without bundling a webfont.

**Why 14px body (not 16):** popup is a confined surface; 14px is the de-facto Chrome extension popup body convention (Discord settings, 1Password, Bitwarden, GitHub Notifier). 16px would force ~30% fewer characters per textarea line and break the 360–400px width contract. Heading is one step up at 16px to maintain a visible hierarchy. Phase 1 used `text-base` (16px) for heading — Phase 2 narrows the type scale to 3 ladder steps and demotes body to 14px.

**Why 2 weights (not 3):** regular (400) for both body prose and field labels — labels are distinguished from body purely by smaller size (`text-xs`) and muted color (`text-slate-500 dark:text-slate-400`), no weight escalation needed. Semibold (600) is reserved for state headings only. No bold tier — popup never has display-class type.

## Component Inventory

Native HTML primitives only — every component below maps to a single tag with Tailwind classes. No custom React components beyond `App.tsx`.

| Component | HTML element | Phase 2 specifics |
|-----------|--------------|-------------------|
| Editable text field | `<textarea>` | always-on edit (D-21); auto-resizes via `field-sizing-content` (Tailwind v4 native) with `min-h` floor; `aria-label` from `t('capture.field.<name>')`; `spellcheck={false}` to suppress red squiggles in URLs and code in `content`. |
| Read-only field (`url`, `create_at`) | `<output>` | inside a labeled wrapper `<div>`; selectable text; `font-mono` for `url` (so domain/path read accurately) and `font-sans` for `create_at`. No copy button (deferred). |
| Field label | `<label>` (when wrapping) or `<span class="sr-only">` (when textarea has `aria-label` only) | every textarea must EITHER have a visible `<label htmlFor>` OR a `aria-label` — not both, not neither. **This phase chooses visible `<label>`** for all 3 editable fields and visible `<label>` for the 2 read-only fields. Visible labels eliminate the a11y ambiguity entirely. |
| Loading skeleton | `<div role="status" aria-busy="true" aria-live="polite">` containing 5 `<div>` blocks shaped like the success-state field stack | one block per upcoming field; pulse animation via `animate-pulse`; `aria-label` from `t('capture.loading.label')` |

## Interaction States

### State machine

```
popup mount
  ↓
  [LOADING]  ← initial; signals: snapshot=null, error=null
  ↓ (RPC settles)
  ├── ok=true  → [SUCCESS]
  ├── ok=false, code=RESTRICTED_URL    → [EMPTY:restricted]
  ├── ok=false, code=EXTRACTION_EMPTY  → [EMPTY:noContent]
  └── ok=false, code=EXECUTE_SCRIPT_FAILED → [ERROR:scriptFailed]
```

State transitions are **one-shot**: once a non-loading state is reached, the popup stays there until the user closes the popup. No automatic retry, no state polling. Re-trigger = close popup, click toolbar icon again (= popup remount = fresh `capture.run`).

### LOADING

- Triggered: popup mount, before RPC settles
- Visual: 5-row skeleton matching the eventual SUCCESS layout (1 row title + 1 row url + 1 row description + 1 row create_at + 1 row content placeholder ~96px tall)
- Each skeleton block: `bg-slate-100 dark:bg-slate-800 rounded animate-pulse h-{4|6|24}`
- a11y: outer container `role="status" aria-busy="true" aria-live="polite"` + visually-hidden text `<span class="sr-only">{t('capture.loading.label')}</span>`
- **Skeleton appears immediately** (no `setTimeout` delay). On fast pages the RPC may resolve in <50ms, in which case the skeleton flashes for 1 frame — that flash is acceptable. On chrome:// pages the SW returns RESTRICTED_URL synchronously after `tabs.query`, so skeleton frame is similarly brief.
- **No timeout / no spinner fallback.** If the RPC genuinely hangs (which CONTEXT does not allow — handler always resolves), Phase 2 does not paper over a Phase 2 bug.

### SUCCESS

- 5 fields rendered top-to-bottom in this exact order:
  1. **Title** — editable `<textarea>` (1 row floor, autosize), label `t('capture.field.title')`
  2. **URL** — read-only `<output>`, label `t('capture.field.url')`, `font-mono text-xs` so long URLs wrap naturally
  3. **Description** — editable `<textarea>` (2-row floor, autosize), label `t('capture.field.description')`
  4. **Captured at** — read-only `<output>`, label `t('capture.field.createAt')`, formatted via `Intl.DateTimeFormat(navigator.language, { dateStyle: 'medium', timeStyle: 'short' })` from the ISO-8601 string
  5. **Content** — editable `<textarea>` (6-row floor, autosize, max-height capped via popup container scroll), label `t('capture.field.content')`
- Editing: typing into any textarea writes to the matching Preact signal (`titleSig`, `descriptionSig`, `contentSig`); read-only fields never bind to signals
- Tab order: title → description → content (read-only `<output>` is **not** in tab order; achieved via `<output>` being a non-focusable element by default and not adding `tabIndex`)
- Focus styling: every textarea on `:focus-visible` shows `ring-2 ring-sky-600 dark:ring-sky-400 ring-offset-2 ring-offset-white dark:ring-offset-slate-900` + replaces resting `border-slate-200 dark:border-slate-700` with `border-sky-600 dark:border-sky-400`
- Resting textarea border: `border border-slate-200 dark:border-slate-700 rounded-md`
- Textarea padding: `px-3 py-2` (12px / 8px — both 4-multiples)
- Textarea typography: inherits body 14/400/1.5

### EMPTY (variants: `restricted`, `noContent`)

- Triggered: `RESTRICTED_URL` or `EXTRACTION_EMPTY`
- Visual: vertically centered stack inside the same `<main class="p-4">` container
  - Icon (24×24px inline SVG, `text-slate-500 dark:text-slate-400`) — info-circle glyph for `noContent`, lock-closed glyph for `restricted`
  - Heading: 16/600/1.3, color body-text — `t('capture.empty.{variant}.heading')`
  - Body: 14/400/1.5, color muted — `t('capture.empty.{variant}.body')`. Body copy contains an inline `<span class="text-sky-600 dark:text-sky-400">` wrapping the noun phrase referring to the action icon (the only inline-accent-text use in the popup); see Copywriting Contract for the three-key split (`.before` / `.icon` / `.after`).
- Layout: `gap-2` (8px) between icon → heading; `gap-1` (4px) heading → body; outer `<section class="flex flex-col items-center text-center py-8 gap-2">`
- a11y: `role="status" aria-live="polite"`; icon is decorative (`aria-hidden="true"`)

### ERROR (`scriptFailed`)

- Triggered: `EXECUTE_SCRIPT_FAILED`
- Visual: identical layout to EMPTY but
  - Icon: alert-triangle glyph, `text-red-600 dark:text-red-400`
  - Heading: 16/600/1.3, color `text-red-600 dark:text-red-400` — `t('capture.error.scriptFailed.heading')`
  - Body: 14/400/1.5, color muted — `t('capture.error.scriptFailed.body')`. Body explicitly tells the user to retry via the toolbar icon (closing + reopening popup re-triggers RPC), with the noun phrase 工具栏图标 / "toolbar icon" wrapped in the inline-accent span.
- a11y: `role="alert" aria-live="assertive"` (note: stronger than empty's `polite`, so screen reader announces as soon as the state lands).

### Retry affordance — validation of CONTEXT recommendation

CONTEXT 02-CONTEXT.md "Claude's Discretion" recommends "no dedicated retry button — repeat = close popup + click action again". The `<critical_constraints>` block in this researcher's brief instructs me to **validate this OR propose a justified deviation**.

**Decision: validated, no retry button.** Rationale:

1. **Semantic alignment with Phase 1 D-09:** popup mount-on-action-click already IS the trigger event. Adding a retry button creates two re-trigger paths, splitting the user's mental model.
2. **No state to preserve across retry:** D-22 forbids draft persistence in Phase 2. There is literally nothing a retry button could "save" the user — close-and-reopen yields the same UX as button-click-retry.
3. **Bundle / surface area:** a button = focusable element + click handler + busy state synchronization with Phase 2's `loading` state. The cost is zero benefit because the toolbar icon is *always* visible to the user and is the canonical action surface.
4. **Phase 3 will introduce real buttons (Send, history dropdown).** A Phase 2 retry button would be the only button in the popup, and it would disappear in Phase 3 when the popup grows — a transient affordance that confuses muscle memory.

The error / empty body copy MUST explicitly tell the user to "click the toolbar icon to retry" so the retry path is **discoverable**, not implicit. This is the single non-negotiable a11y obligation that replaces the missing button.

### Copywriting principles applied

- **Plain language.** No technical terms (no "executeScript", no "DOMPurify", no "Readability", no "ISO-8601"). The user does not know — and should not need to know — the implementation.
- **Imperative + actionable.** Empty/error body always ends with a sentence telling the user *exactly what to do next* ("Open a regular website, then click the toolbar icon again"). Never a dead-end "Sorry, something went wrong."
- **Verb-first action references.** "Click the toolbar icon" / "点击工具栏图标". Never "you may wish to consider clicking" or other softeners.
- **No exclamation marks, no emoji.** Functional tone. (Developer profile: no emojis unless explicitly requested.)
- **No "please" in en, "请" sparingly in zh_CN.** zh_CN uses "请" only in the error retry sentence where formal politeness is conventional; en uses imperative directly.
- **Pluralization not required this phase.** No `$COUNT$`-style placeholders in any capture.* string (CAP-* fields are singular). Phase 1 `popup_hello` substitution stays untouched.
- **Mirror length:** zh_CN strings are 1.5–2.0× shorter than en in character count but visually similar in width because zh chars are wider. `min-w-[360px]` guarantees both languages fit on 1–2 lines for headings and 2–3 lines for body copy.

### Primary CTA

**There is no CTA in Phase 2.** This is intentional: the popup performs its job automatically on mount and presents data for in-place editing only. The next CTA — Send — lives in Phase 3 (DSP-05).

The "implicit CTA" surrogate in error / empty states is the **toolbar icon itself**, referenced verbatim in the body copy. This is a deliberate UX inversion: instead of "give the user a button", we say "the affordance you already used is the same affordance you use to retry".

### Destructive actions

**None in this phase.** D-22 forbids any storage write; therefore there is no data to clear, reset, or delete. Destructive color (`red-600`) is used purely as a semantic signal in the error state heading + icon, not for any user-initiated destructive action.

## Accessibility Contract

| Concern | Specification |

## Component / Pattern Notes for Executor

These are **non-design** notes the executor needs to implement the contract correctly. Listed here so the planner doesn't re-discover them.

1. **Auto-grow textareas** — use Tailwind v4 native `field-sizing-content` utility. Falls back gracefully on Chrome <129; Phase 1 verify-manifest already locks Chrome MV3 baseline so this is safe. If `field-sizing-content` rendering glitches in dev, plan a 1-line `onInput` height-sync as escape hatch — but try the native utility first.
2. **Date formatting** — `Intl.DateTimeFormat(navigator.language, { dateStyle: 'medium', timeStyle: 'short' })` reads the SW-generated ISO-8601 string and formats per locale. No date-fns / dayjs library — pure native Intl.
3. **Inline accent spans** — emit as JSX `<span class="text-sky-600 dark:text-sky-400">{noun}</span>`, **not** via i18n template substitution with HTML. The substitution mechanism here is JSX composition: `t('capture.empty.restricted.body.before') + <span>{t('capture.empty.restricted.body.icon')}</span> + t('capture.empty.restricted.body.after')` — three keys per body string (see Copywriting Contract for the explicit `.before` / `.icon` / `.after` sub-key table). **Reject** putting raw HTML inside a single i18n key (XSS surface; PITFALLS §11). Implementation note for planner: when authoring zh_CN/en messages, split the sentence into three parts (`before`, `icon`, `after`) keyed under each empty/error variant — so the JSX `<span>` lives in code, the text segments live in YAML.
4. **Skeleton dimensions** — must approximate the success layout to prevent layout shift. Recommended: `[h-4 w-1/3]` for label, `[h-9 w-full]` for title row, `[h-9 w-full]` for url row, `[h-4 w-1/4]` for createAt label, `[h-9 w-full]` for createAt row, `[h-24 w-full]` for content row. Rounding: `rounded` (4px) consistently.
5. **No `innerHTML`** — PITFALLS §安全错误 #1. Every text rendering goes through Preact text nodes. The captured `content` Markdown is rendered as **raw text inside the textarea** (`textarea.value = content`), not as parsed Markdown. Phase 2 has no Markdown preview; if a future phase wants one, that's a separate UI contract.
6. **Read-only fields use `<output>`** — semantically correct (it's a calculation result), is non-focusable by default, and is announced by AT as "output". Do not use disabled `<input>` — that announces as "disabled, edit text" which is wrong.
